## 三、逐技术评估

### 3.1 前端核心

| 技术 | 版本 | 选型理由 | 替代方案 | 评估 |
|------|------|----------|----------|------|
| **Vue 3** | 3.5+ | Composition API + `<script setup>` 简洁高效；生态成熟；社区活跃 | React 19 / Svelte 5 | 团队最熟悉 Vue，选型合理 |
| **TypeScript** | 5.x | 前后端统一类型体系，类型安全 | -- | 必须保留 |
| **Vite** | 6.x | HMR 极速，Vue 官方推荐，插件丰富 | Webpack / Turbopack | 最优选择 |
| **Pinia** | 2.x | Vue 3 官方状态管理，TypeScript 原生支持 | Vuex 4 | 无争议 |
| **Vue Router** | 4.x | 官方路由方案，懒加载/守卫开箱即用 | -- | 无争议 |

### 3.2 前端辅助

| 技术 | 选型理由 | 评估 |
|------|----------|------|
| **ofetch** | 比 Axios 轻 60%，原生 TS 类型推断，Nuxt 团队维护 | 替代 Axios 合理，减少依赖体积 |
| **@vueuse/motion** | 淘汰/翻牌/结算动画声明式 API，支持 spring/tween | 比手写 CSS animation 更可维护 |
| **vite-plugin-pwa** | Workbox 集成，离线缓存 + 可安装一行配置 | PWA 加分项的关键依赖 |
| **Web Speech API** | 浏览器原生 TTS，零依赖 | 初期方案，后期可切后端 TTS |

> **不引入 UI 组件库的理由**：本项目交互场景独特（发言气泡、投票面板、AI 打字机、角色揭示动画），通用组件库无法满足定制需求，自建组件更灵活且体积更小。

### 3.3 后端核心

| 技术 | 版本 | 选型理由 | 替代方案 | 评估 |
|------|------|----------|----------|------|
| **Node.js** | 22 LTS | 异步 I/O 天然匹配 WS + SSE 流式；前后端统一语言 | Java 21 / Go 1.23 / Python 3.12 | 选型合理（详见 3.6） |
| **Fastify** | 5.x | 性能优于 Express 3-5x；原生 TS；插件生态；Schema 校验内置 | Express / Koa / Hono | 优秀选择 |
| **TypeScript** | 5.x | 类型安全；与前端共享类型定义 | -- | 必须保留 |

### 3.4 后端关键依赖

| 技术 | 选型理由 | 评估 |
|------|----------|------|
| **@fastify/websocket** | Fastify 原生 WS 插件，路由级注册，自动升级 | 比 socket.io 轻量，无额外协议 |
| **xstate v5** | 游戏状态机可视化/自文档化；Actor 模型适合并发房间；TypeScript 类型推断完美 | 比手写 switch-case 可维护性高一个量级 |
| **Drizzle ORM** | 类 SQL 语法直观；体积小(7KB)；drizzle-zod 自动生成校验 Schema | 比 Prisma 轻量，比 TypeORM 现代 |
| **Zod** | 与 Fastify Schema 集成；与 Drizzle 联动；前后端可共用 | 校验层标准选择 |
| **Pino** | Node.js 最快的日志库；JSON 结构化；Docker 友好 | 生产环境日志标配 |
| **bcrypt** | 密码哈希工业标准 | 不可替代 |
| **jsonwebtoken** | JWT 签发/验证 | 无争议 |

### 3.5 数据层

| 技术 | 版本 | 选型理由 | 评估 |
|------|------|----------|------|
| **PostgreSQL** | 16 | 成熟稳定；JSONB 支持（游戏事件存储）；Drizzle 原生支持 | 比 MySQL JSONB 操作更强；适合游戏数据结构多变的特点 |
| **Redis** | 7 | 房间状态内存缓存；Sorted Set 排行榜天然适配；Pub/Sub 跨进程通信 | 游戏服务标配 |

### 3.6 运行时选型对比：Node.js vs Java

| 维度 | Node.js (TypeScript) | Java (Spring Boot) | 本项目倾向 |
|------|---------------------|--------------------|------------|
| 前端类型共享 | 直接 import 同一份 `.ts` 类型 | 需 OpenAPI/Protobuf 生成 | **Node.js 胜** |
| WebSocket | 事件驱动模型原生匹配 | Stomp 协议层冗余 | **Node.js 胜** |
| SSE 流式 | Stream API 原生 pipe | 需额外处理 | **Node.js 胜** |
| JSON | 零成本 | 序列化开销 | **Node.js 胜** |
| 状态机 | xstate 是 TS 原生库 | 需手写或引入第三方 | **Node.js 胜** |
| 单人全栈 | 心智模型统一 | 上下文切换 | **Node.js 胜** |
| 并发能力 | 事件循环百级并发无压力 | 多线程万级并发 | 持平（本项目量级不到） |
| 启动速度 | <1s | 5-10s | **Node.js 胜** |
| 镜像体积 | ~150MB | ~250MB+ | **Node.js 胜** |

> **结论**：本项目 JS/TS 全栈统一语言带来的收益远超 Java 在极端并发场景的性能优势。

### 3.7 AI 服务层

| 技术 | 选型理由 | 评估 |
|------|----------|------|
| **Dify Platform** | 可视化工作流编排；支持 SSE 流式输出；内置 RAG/Agent；私有部署可控 | 符合需求的 AI 中台选择 |
| **工作流模式** | 4 个独立 App，各司其职 | 职责清晰，可独立调优 |

### 3.8 运维/部署

| 技术 | 选型理由 |
|------|----------|
| **Docker** | 环境一致性；一行命令启动全栈 |
| **Docker Compose** | 编排 PG + Redis + App，适合小规模部署 |
| **Nginx** | 反向代理 + SSL + WS 升级 + 静态资源 |
| **GitHub Actions** | 免费 CI/CD，自动构建镜像 + 部署 |

## 四、未选用的技术及原因

| 未选用技术 | 常见用途 | 本项目不选用的原因 |
|------------|----------|--------------------|
| socket.io | WebSocket 封装 | 协议冗余（自动降级长轮询不必要）；Fastify 原生 WS 更轻量 |
| Prisma | ORM | 体积大(6MB+)；迁移工具重；Drizzle 更轻量且 SQL 直觉更强 |
| Express | HTTP 框架 | 性能不如 Fastify；TS 支持不如 Fastify 原生 |
| MySQL | 关系型数据库 | 缺少 JSONB；PG 的 JSONB 更适合游戏事件/状态存储 |
| MongoDB | NoSQL 数据库 | 游戏结算数据有强关联关系，关系型更合适 |
| GraphQL | API 查询语言 | 本项目 API 结构简单，REST + WS 已足够，额外复杂度不划算 |
| Kafka/RabbitMQ | 消息队列 | 初期没有异步任务解耦需求，Redis Pub/Sub 足够跨进程通信 |
| Kubernetes | 容器编排 | 单体应用不需要集群调度，Docker Compose 足够 |
| Next.js / Nuxt | 全栈框架 | SSR 对 SPA 游戏类型无收益；Vite + Fastify 分离部署更灵活 |
| Tailwind CSS | 原子化 CSS | 游戏 UI 组件定制化程度高，CSS Variables + scoped 更可控 |
| shadcn-vue | Vue 组件库 | 重度依赖 radix-vue，游戏 UI 场景不匹配 |

## 五、风险点与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| Dify 服务不可用 | AI 发言/投票/词语生成全部挂掉 | 词语缓存池(L1)；本地模板降级(L2)；兜底发言(L3) |
| WebSocket 断连 | 游戏体验中断 | 指数退避重连 + game_state 快照恢复 |
| Redis 内存溢出 | 房间状态丢失 | 设置 maxmemory + LRU 驱逐；游戏结束后异步落库 PG |
| 单进程 Node.js 瓶颈 | 百级并发后 CPU 打满 | PM2 Cluster 多进程 + Redis Pub/Sub 状态同步 |
| AI 发言延迟 | 玩家等待焦虑 | 流式推送打字动画；15s 超时降级；loading 提示 |
