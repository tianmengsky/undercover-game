# RAG核心知识

- RAG：检索增强生成
- 打分  排序  召回10条  重排  2条

# 一、全文检索（关键词检索）

## 1.原理

全文检索 = **电脑 Ctrl+F 查找**

只匹配**字面文字**，不理解语义、不理解近义词。

你写什么字，它就找什么字，字不一样就搜不到。

## 2. 工作流程

输入关键词 → 匹配包含该文字的文档 → 计算匹配分数 → 排序返回

## 3. 实战案例（谁是卧底词库）

词库数据：

1. 馒头-面包（难度：简单）

2. 薯片-锅巴（难度：适中）

3. 毛巾-浴巾（难度：适中）

4. 钢笔-毛笔（难度：困难）

搜索关键词：**适中**

结果：只返回两条带“适中”的词库数据。

缺点：如果搜“零食”，虽然薯片、锅巴是零食，但文字不匹配，**完全搜不到**。

## 4. 全文检索打分规则

**核心结论：匹配关键词 ≠ 高分**

很多同学疑惑：**我明明精准匹配到关键词了，为什么分数很低？**

因为全文检索的分数由**三个维度综合计算**，不是搜到就满分。

### 维度1：词频（关键词出现次数）

同一个关键词，**出现次数越多，分数越高**。

例子：

文本A：难度：适中（出现1次）→ 分数一般

文本B：难度：适中、整体难度：适中、本局难度：适中（出现3次）→ 分数更高

### 维度2：文本长度（关键词占比）

同样只出现1次关键词：

- 文本越短 → 关键词占比越高 → 分数越高
- 文本越长 → 关键词占比越低 → 分数越低

例子：一大段话最后只带一个“适中”，分数会很低。

### 维度3：词语稀有度（最关键原因）

如果一个词在**所有数据里到处都有**，它就没有区分度，系统会压低分数。

- 高频通用词（简单/适中/困难/难度）→ 全员都有 → **整体分数普遍偏低**
- 冷门独有词（锅巴、浴巾）→ 只有少数数据有 → 匹配就高分

## 5. 解答核心疑惑

**为什么难度关键词完全匹配，分数却不高？**

1. 每条数据难度词只出现1次（词频低）
2. 简单、适中、困难是全局高频标签（稀有度极低）
3. 所以：**百分百命中，分数依然偏低，属于正常机制，不是bug**

## 6. 全文检索优缺点

优点：精准、稳定、速度快，适合标签筛选

缺点：不懂语义，只会字面匹配，打分对通用标签不友好

# 二、向量检索（语义检索）

## 1. 原理

向量检索 **不看文字，只看意思**。

类比：别人描述“白白软软、早餐主食”，你不用看到“馒头”二字也能猜到。

## 2. 什么是向量化

电脑看不懂文字，只能看懂数字。

系统会把每一段话、每个词语，转换成一串**专属数字标签（向量）**。

**意思越相近，数字越相似；意思越远，数字差距越大。**

例：

- 馒头、面包 → 向量很接近（都是主食）
- 馒头、薯片 → 向量差距很大

## 3. 向量检索打分逻辑

将用户查询的向量 和 库存向量 **比对相似度**：

向量越接近 → 相似度分数越高 → 语义越匹配

## 4. 实战案例

搜索：**早餐常吃的松软主食**

全文检索：搜不到，因为没有字面关键词

向量检索：理解语义，精准召回「馒头、面包」数据

## 5. 优缺点

优点：懂语义、支持模糊搜索、支持描述匹配

缺点：不懂业务规则，容易召回语义相近但标签不对的数据

# 三、混合检索

## 1. 原理

**全文检索 + 向量检索 结合**

取长补短：

- 全文检索：负责**精准过滤、标签筛选**（难度、分类）
- 向量检索：负责**语义匹配、相似度判断**

## 2. 工作流程

**先过滤、后语义**

1. 全文检索：先按难度/标签筛出合规数据
2. 向量检索：在合规数据中做语义相似度打分
3. 融合双方分数，得到最终综合得分
4. 重排输出最优结果

## 3. 实战案例

需求：找出**难度适中、属于零食类**的词语对

1. 全文检索：筛选出所有「难度适中」的数据（薯片锅巴、毛巾浴巾）
2. 向量检索：在里面匹配“零食”语义
3. 最终精准返回：薯片 & 锅巴

## 4. 优缺点

优点：既精准、又智能，是 AI 知识库、游戏词库的最优方案

缺点：逻辑比单一检索复杂

# 四、三种检索总结

- **全文检索**：Ctrl+F 字面匹配，适合标签筛选，分数受词频、长度、稀有度影响
- **向量检索**：语义匹配，靠向量相似度打分，适合模糊搜索、描述匹配
- **混合检索**：先关键词过滤、后语义排序，项目最稳、最常用

# 五、检索工作流程

不管是全文、向量、混合检索，线上服务都遵循 **「召回 → 打分 → 重排」** 三段式架构

### 阶段 1：召回（粗排 / 粗筛选）

#### 1. 目标

从**全量数据集**中，快速选出一批候选数据，**牺牲部分精度，换取极致速度**。

#### 2. 执行规则

- 数量控制：召回数量 > 最终输出数量

  例：最终要输出 10 条 → 先召回 20~50 条候选；

- 技术选型：

  - 全文检索：倒排索引快速匹配召回；
  - 向量检索：向量库索引（IVF、HNSW）快速近似召回；
  - 混合检索：关键词过滤 + 向量快速召回。

#### 3. 特点：只做初步打分、粗略排序。

------

### 阶段 2：统一打分

对召回得到的候选列表，逐条计算**标准相似度得分**：

1. 全文检索：计算 TF-IDF / BM25 分数；
2. 向量检索：计算余弦相似度 / 欧式距离；
3. 混合检索：加权融合得分。

所有候选数据，都会生成一条唯一的「综合得分」。

------

### 阶段 3：重排（精排 / 精筛选）

#### 1. 目标

对召回候选集做**精细化排序、择优筛选**，输出最终结果。

#### 2. 常见重排方案（由简到难）

1. **规则重排**

   直接按检索得分降序排列，取 Top K。

   例：得分从高到低，截取前 10 条作为最终结果。

2. **业务规则二次调整**

   在分数基础上，叠加自定义业务规则：

   - 随机打乱（你需要：每次返回不同词语对）；
   - 过滤重复数据；
   - 按难度、标签优先级调整顺序。

3. **模型重排**

   使用专门的 Rerank 重排模型，对候选列表重新计算相关性，大幅提升精度。

#### 3. 最终输出

重排完成后，截取指定数量数据，返回给前端 / 后端调用。

### 完整链路总结

```
原始数据预处理（建索引/向量化）` → `用户查询` → `召回（粗捞候选）` → `逐条打分` → `重排（精细排序）` → `最终结果输出
```

## 六、三类检索 横向对比表

| 对比维度 |      全文检索      |       向量检索       |          混合检索           |
| :------: | :----------------: | :------------------: | :-------------------------: |
| 核心能力 |   关键词字面匹配   | 语义理解、相似度匹配 |      关键词 + 语义兼顾      |
| 底层结构 |      倒排索引      |  向量 + 向量数据库   |     倒排索引 + 向量索引     |
| 打分依据 |   TF-IDF / BM25    | 余弦相似度、欧式距离 |        加权融合得分         |
| 检索速度 |        极快        |   较快（依赖索引）   |            较快             |
| 语义支持 |       不支持       |        强支持        |           强支持            |
| 开发难度 |         低         |          中          |             高              |
| 推荐场景 | 精准筛选、字段查询 |  语义问答、模糊检索  | AI 知识库、综合检索（首选） |
