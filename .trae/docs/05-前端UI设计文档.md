# AI谁是卧底大乱斗 — 前端 UI 设计文档（v2 - 多人联网改造）

> **变更摘要**：将单人模式改造为多人联网模式。大厅改为房间列表 + 房间等待页，支持创建/加入房间、断线 AI 托管、玩家退出/重连。UI 风格与 v1 保持一致。
>
> 依赖：socket.io（已于 2026-06-09 完成迁移）

---

## 零、多人联网核心流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 首页     │───→│ 房间列表  │───→│ 房间等待  │───→│ 游戏进行  │
│ Home     │    │ /rooms   │    │ /room/:id│    │ /game/:id│
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                     │  ↑            │  ↑           │  ↑
                     │  │            │  │           │  │
              创建房间  返回    加入房间  返回    退出游戏  结算
              POST/games 房间    socket.io  房间   回到房间   /result
```

### 0.1 关键状态事件

| 场景 | socket.io 事件（服务端→客户端） | UI 响应 |
|------|-------------------------------|---------|
| 新房间创建 | `room_list` 广播 | 房间列表刷新 |
| 玩家加入 | `player_joined`（房间页）/ `room_updated`（列表） | 槽位填充、人数更新 |
| 玩家退出 | `player_left` | 槽位清空、人数更新 |
| 游戏中断线 | `player_disconnected` → 30s 倒计时 → `ai_takeover` | 显示"中断连接"标签，超时后变 AI 标签 |
| 玩家重连 | `player_reconnected` | AI 托管标签消失，恢复玩家标识 |
| 房间无真人 | `room_deleted` | 弹窗提示 + 跳转房间列表 |
| 房间满员 | `room_list` 更新 | 列表中房间状态显示"游戏中"或隐藏 |

### 2.2 设计细节

| 区域 | 规范 |

### 2.3 交互

| 操作 | 触发 | 行为 |
|------|------|------|

| 加入房间 | 点击 [加入] | POST `/api/rooms/join`（含 roomCode）→ 拿到 gameId → socket.io 连接 → 跳转 `/room/:roomId` |

| 房间号不存在 | join 失败 | Toast 红色 "房间不存在或已满" |

### 3.2 设计细节

**槽位卡片（6 个，2 行 × 3 列网格）：**

| 槽位类型 | 视觉 | 操作 |
|----------|------|------|
| **当前用户** | 蓝色边框 `#4FACFE`，头像 + 昵称 + "(你)" 标签，房主显示 `⭐` | 可修改昵称（内联编辑），不可移除 |
| **其他人类玩家** | 绿色边框 `#2ECC71`，头像 + 昵称 + "(已加入)" | 不可操作（仅房主可踢人） |
| **AI 槽位（已设定人设）** | 灰色边框，AI 头像 + 人设名称 + 人设下拉 | 可点击下拉切换人设 |
| **空槽位（无 AI/无玩家）** | 虚线边框，灰色文字 "等待加入" | 房主可点击切换为 AI 填充，或等待真人加入 |

**人设下拉菜单：**
- 弹出卡片式，每项显示人设图标 + 名称 + 一行描述
- 选项：`默认`、`幽默达人`、`逻辑大师`、`萌新小白`、`文艺青年`、`暴躁老哥` 以及从工坊选用的自定义人设
- 选中项高亮主蓝色

**词语设置区域：**
- 与 v1 大厅设计完全一致，保持不变

**底部按钮（仅房主可见）：**
- `[开始游戏]` 大号按钮，至少 2 名玩家可点击
- 不够 2 人时灰色 disabled + tooltip "至少需要 2 名玩家"
- 点击后先自动补全空位为默认 AI → POST `/api/games/:id/start` → 所有玩家跳转 `/game/:gameId`

### 3.3 房间创建者（房主）权限

| 操作 | 方式 |
|------|------|
| 修改自己昵称 | 点击自己槽位的昵称区域，内联编辑 |
| 踢出玩家 | 右键 / 长按某玩家槽位 → "移除玩家" → 确认弹窗 → `kick_player` socket.io 事件 |
| 设置 AI 人设 | 点击 AI 槽位的下拉菜单 |
| 将 AI 槽位改为空位（等待真玩家） | 点击 AI 槽位的 `[× 移除 AI]` |
| 修改词语 | 与 v1 相同 |
| 开始游戏 | 底部按钮 |

### 3.4 非房主玩家界面

- 无法修改词语、无法踢人、无法开始游戏
- 可修改自己昵称
- 可查看所有槽位信息
- 底部显示 "等待房主开始游戏..." + 旋转加载点动画

### 3.5 实时交互

| 事件 | trigger | UI 更新 |
|------|---------|---------|
| 新玩家加入 | socket.io `player_joined` | 对应空位变为玩家卡片（绿色边框），人数 +1 |
| 玩家离开 | socket.io `player_left` | 对应槽位恢复为 AI 或空位，人数 -1 |
| 被踢出 | socket.io `kicked_from_room` | Toast 提示 + 跳转 `/rooms` |
| 房主离开 | socket.io `player_left` + `host_changed` | 房主标识自动转移给下一个人，"你不是房主了" Toast |
| 房间满员 | 最后一个空位被占 | 房间列表中该房间变为"游戏中" |
| 词语变更 | socket.io `words_updated` | 词语显示区刷新 |

## 六、新增 socket.io 事件 与 REST API

> **架构决策**：房间列表浏览使用 REST API（避免 socket.io auth 中间件强制 gameId 的冲突），房间内交互使用 socket.io。

### 6.1 REST API（新增）

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET | `/api/rooms` | 获取房间列表 | 是 |
| POST | `/api/rooms/join` | 通过房间号加入，返回 gameId | 是 |

**POST /api/rooms/join 请求：**
```json
{ "roomCode": "A7K2M9", "nickname": "柯南" }
```

**响应：**
```json
{
  "code": 0,
  "data": {
    "gameId": "uuid-xxx",
    "roomId": "uuid-xxx",
    "roomCode": "A7K2M9",
    "roomName": "推理大师的房间",
    "yourSlotIndex": 0
  }
}
```

> **设计说明**：客户端拿到 `gameId` 后，通过 socket.io 连接时在 `auth` 中传入，完成房间加入。房间创建仍使用现有 `POST /api/games`，需扩展为支持 `roomCode` 和 `roomName` 参数，且创建时 slot 全部为空（不传 players 数组）。

**GET /api/rooms 响应：**
```json
{
  "code": 0,
  "data": {
    "rooms": [
      {
```

### 6.2 房间内 socket.io 事件

| 方向 | 事件名 | payload | 说明 |
|------|--------|---------|------|
| C→S | `set_persona` | `{ slotIndex, persona }` | 房主设置 AI 人设 |
| C→S | `set_words` | `{ civilianWord, undercoverWord }` | 房主设置词语 |
| C→S | `kick_player` | `{ targetUserId }` | 房主踢人 |
| C→S | `leave_room` | `{}` | 离开房间（游戏前） |
| C→S | `leave_game` | `{}` | 游戏中退出 |
| C→S | `rejoin_game` | `{}` | 重新加入进行中的游戏 |
| S→C | `room_updated` | `{ players: SlotInfo[], hostId }` | 房间状态变更 |
| S→C | `player_joined` | `{ userId, nickname, slotIndex }` | 新玩家加入 |
| S→C | `player_left` | `{ userId, nickname, slotIndex }` | 玩家离开 |
| S→C | `host_changed` | `{ newHostId, newHostName }` | 房主转移 |
| S→C | `kicked_from_room` | `{ reason }` | 被踢出 |

### 6.3 游戏中管理事件

| 方向 | 事件名 | payload | 说明 |
|------|--------|---------|------|
| S→C | `player_disconnected` | `{ userId, nickname, slotIndex }` | 玩家断线，30s 倒计时开始 |
| S→C | `ai_takeover` | `{ userId, slotIndex, nickname }` | AI 接管该玩家 |
| S→C | `player_reconnected` | `{ userId, nickname, slotIndex }` | 玩家重连，恢复控制权 |
| S→C | `room_deleted` | `{ reason }` | 房间被关闭（无真人） |

### 6.4 房间数据结构

```typescript
interface RoomInfo {
  roomCode: string      // 6 位公开房间号
  roomName: string      // 房间名称
  playerCount: number   // 当前人数
  maxPlayers: number    // 固定 6
  status: 'waiting' | 'playing'
  hostName: string      // 房主昵称
}

interface SlotInfo {
  slotIndex: number
  type: 'human' | 'ai' | 'empty'
  nickname?: string
  userId?: string       // 仅 human 有
  persona?: string      // 仅 ai 有
  isHost: boolean
  playerStatus?: 'normal' | 'disconnected' | 'ai_takeover'
}
```

> **后端注意事项**：
> - `GameRoom` 需新增 `roomCode`（6 位字母数字，创建时随机生成，校验唯一）、`roomName` 和 `hostUserId` 字段
> - `speechResolvers` / `voteResolvers` 需改为按 `slotIndex` 索引（如 `${gameId}_${slotIndex}`），支持多人同时等待发言/投票
> - 玩家断线 30s 后需由服务端触发 `ai_takeover` 事件 + 将槽位的 `Player.type` 改为 `'ai'`
> - 当房间内所有人类玩家断开时，服务端广播 `room_deleted` 并清理房间资源
