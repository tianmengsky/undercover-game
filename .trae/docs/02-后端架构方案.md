## 一、项目概述

### 1.1 职责定位
后端服务负责用户认证、游戏房间管理、游戏状态机流转、AI 发言编排（对接 Dify）、排行榜数据持久化。通过 WebSocket 与前端实时通信，通过 REST API 提供非实时数据。

### 1.2 技术选型

| 类别 | 技术 | 说明 |
|------|------|------|
| 运行时 | Node.js 20+ (LTS) | 异步 I/O 友好，WebSocket 生态成熟 |
| 框架 | Fastify 5.x | 高性能 HTTP，原生 TS 支持，插件生态 |
| WebSocket | `socket.io` | 游戏信令，自动重连 + Room 管理 + 心跳 |
| 语言 | TypeScript 5.x | 类型安全 |
| 数据库 | PostgreSQL 16 + Redis 7 | PG 持久化，Redis 缓存+房间状态 |
| ORM | Drizzle ORM | 类型安全，轻量级 |
| 认证 | JWT (access + refresh token) | 无状态认证 |
| AI 调用 | Dify API (HTTP + SSE 流式) | 词语生成、AI 发言生成 |
| TTS | 腾讯云 / 阿里云 TTS | 语音合成（按需，可先不分） |
| 部署 | Docker + Docker Compose | 容器化 |
| 日志 | Pino | 结构化日志 |

## 四、游戏状态机（核心逻辑）

### 4.1 状态流转

```
CREATED → WAITING → ROLE_ASSIGN → [SPEAKING ⇄ VOTING] → RESULT → FINISHED
                                                    ↑          │
                                                    └──────────┘ (多轮循环)
```

### 4.2 各阶段详解

#### WAITING — 等待开始
- 玩家配置角色名、词语
- 触发：玩家点击「开始游戏」→ 校验 6 个角色已就绪 → 进入下一阶段

#### ROLE_ASSIGN — 身份分配
- 随机指定 1 人为卧底，其余为平民（支持 1 卧底 / 2 卧底两种模式）
- 玩家查看自己的词语（仅自己可见，WS 单独推送）
- 自动进入 SPEAKING
- **多人支持**：人类玩家数量不限于 1，支持 1~6 人混合 AI

#### SPEAKING — 发言阶段
- 按座位顺序轮流发言
- 人类玩家：等待 WebSocket 输入 → 广播发言
- AI 玩家：调用 Dify API（带人设 System Prompt）→ 流式返回 → 广播
- 每轮所有存活玩家发言完毕后 → 进入 VOTING

#### VOTING — 投票阶段
- 所有存活玩家投票选出一人淘汰
- AI 投票逻辑：服务端分析本局所有发言，`Promise.all()` 并行调用 Dify 决策票谁（5 个 AI 并发，总耗时 ≈ 单次请求延迟）
- 平票 → 无人淘汰，进入下一轮
- 投票结束 → 淘汰得票最多者 → 判断胜负

#### 胜负判断
- 卧底被淘汰 → **平民胜利**
- 场上只剩 2 人（1平民+1卧底 或 卧底存活到最后）→ 检查卧底是否存活：
  - 卧底存活 → **卧底胜利**
  - 否则继续
- 达到最大轮次（5轮）卧底仍存活 → **卧底胜利**

#### RESULT → FINISHED
- 结算奖励、更新统计数据
- 房间保留 5 分钟后销毁

### 4.3 游戏调度器 (`game.orchestrator.ts`)

调度器是游戏流程的实际驱动者，负责协调状态机流转、AI 调用和人类输入等待。

**触发时机：** `POST /api/games/:id/start` 返回 HTTP 响应后异步启动，不阻塞请求。

**主循环流程：**

```
runGameLoop(gameId)
  ├── broadcast: game_started
  │
  ├── while (winner == null)
  │   ├── runSpeakingPhase(gameId)
  │   │   └── for each 存活玩家（按座位顺序）:
  │   │       ├── AI: generateSpeech() SSE 流式 → broadcast ai_speech_start/chunk/end
  │   │       └── 人类: broadcast request_speech → await waitForHumanSpeech() → broadcast player_speech_broadcast
  │   │
  │   ├── runVotingPhase(gameId)
  │   │   └── for each 存活玩家:
  │   │       ├── AI: generateVote() → broadcast vote_cast
  │   │       └── 人类: broadcast request_vote → await waitForHumanVote() → broadcast vote_cast
  │   │
  │   ├── runEliminationPhase(gameId)
  │   │   ├── calculateElimination() — 计票淘汰，平票无人淘汰
  │   │   ├── checkWinner() — 判胜逻辑
  │   │   └── broadcast: player_eliminated 或 elimination_tie → game_over
  │   │
  │   └── 下一轮: currentRound++, 清空 speeches/votes
  │
  └── finally: markLoopDone
```

**人类输入等待机制：**
- `roomManager.waitForHumanSpeech(gameId, timeout)` 返回 Promise
- WS 收到 `player_speech` → `roomManager.resolveHumanSpeech()` → Promise resolve
- 超时自动跳过该玩家

**关键设计决策：**
- 不使用 xstate 的 `interpret()` 运行时，而是通过 `roomManager` 直接管理状态（简化开发，避免状态同步问题）
- `game.engine.ts` 保留作为状态流转的参考设计和测试用
- 调度器运行期间通过 `runningLoops` 集合防止重复启动

## 五、Dify AI 集成

### 5.1 调用场景

| 场景 | Dify 应用 | 输入 | 输出 | 优先级 |
|------|----------|------|------|--------|
| 词语生成 | `word-generator` | 无/主题词/难度 | 一对相似词语 | 后台批量预生成（**RAG知识库 + LLM选择**，非纯LLM生成） |
| AI 发言 | `undercover-speaker` | 人设、词语、历史发言、游戏上下文 | 一段发言 | 实时（SSE 流式） |
| AI 投票 | `undercover-voter` | 人设、角色、历史发言 | 投票目标 + 理由 | 实时（并行调用） |
| 人设创建 | `persona-creator` | 自然语言描述 | System Prompt | 低频（玩家工坊） |

### 5.2 词语缓存池策略（RAG 知识库方案）

**核心思路：** Dify 知识库作为词库的"主存储"，本地 `word_cache` 表作为二级缓存。词语生成不再依赖纯 LLM 凭空编造，而是从 Dify 知识库中检索真实标注词对。

**Dify 知识库职责：**
- 存储 200+ 对标注好难度和主题的词对（CSV 格式上传）
- 配置混合检索（全文检索按难度/主题过滤 + 向量检索匹配语义）
- 每次词语生成请求：Dify 工作流先知识检索召回 10 对候选 → LLM 随机选 1 对

**本地 `word_cache` 表职责：**
为降低 Dify 调用成本并提升响应速度，采用「预生成 + 缓存」模式：

```
┌──────────┐     定时任务(每小时)     ┌─────────┐
│  Dify    │◄───────────────────────│ Cron    │
│ word-gen │                         │ Job     │
└────┬─────┘                         └────┬────┘
     │ 批量生成                              │
     ▼                                      ▼
┌──────────┐   游戏创建时随机抽取    ┌─────────────┐
│ word_cache│◄────────────────────│POST /games  │
│ (200+对)  │                      │ /words/gen  │
└──────────┘                      └─────────────┘
```

- `word_cache` 表维护最少 200 对词语
- 定时任务检测 `usage_count < 3` 的词语对数量，不足时触发 Dify 批量补货
- 游戏创建时优先从缓存随机抽取，仅当缓存不足时实时调用 Dify
- 难度分级：`1`(日常物品)、`2`(抽象概念)、`3`(专业术语)

### 5.3 AI 发言超时与降级

| 策略 | 超时 | 行为 |
|------|------|------|
| 主路径 | 15s | Dify SSE 流式返回 → 后端逐块转发 |
| 降级 L1 | 15s 超时 | 使用本地模板生成规则发言（模板池 20+ 条） |
| 降级 L2 | L1 也失败 | 发送固定兜底发言（"嗯...让我想想该怎么描述好呢"） |

模板发言示例（逻辑大师人设）：
```
"根据已知信息分析，这个对象具有以下特征：[属性A]、[属性B]，但我不排除[属性C]的可能性。"
```
其中 `[属性A/B/C]` 根据词语从特征库随机填充。

**温度参数按人设动态调整：**
| 人设 | temperature | 说明 |
|------|-------------|------|
| default | 0.7 | 中性 |
| humor | 0.9 | 高创造性 |
| logic | 0.4 | 严谨克制 |
| newbie | 0.95 | 随机跑题 |
| literary | 0.8 | 文艺发散 |
| grumpy | 0.6 | 直接但带刺 |

### 5.4 Dify 工作流设计要点

**AI 发言工作流 (`undercover-speaker`)：**
```
System Prompt: 你正在玩谁是卧底游戏。你的角色是{persona}。
你的词语是{word}，你的身份是{role}。
游戏历史：{game_history}
当前轮次：{round}

请用{persona}的语气，用一句话描述你的词语。
注意：
- 不能直接说出词语
- 如果你是卧底，你不知道其他玩家的词语，请根据其他人的发言推测并模仿
- 发言要符合你的人设风格
```

**AI 投票工作流 (`undercover-voter`)：**
```
System Prompt: 分析本轮所有发言，选出最可能是卧底的人。
你扮演 {persona}。
本轮发言：{round_speeches}
请给出投票对象（玩家名称）和简短理由。
```

### 5.5 调用方式
- 使用 Dify HTTP API (`/v1/chat-messages` 或 `/v1/workflows/run`)
- 发言支持 SSE 流式输出 → 后端转发给前端（打字动画数据源）
- AI 投票：5 个 AI 并行 `Promise.all()` 调用，控制在 5s 内完成
- 超时处理：15s 超时 → L1 降级发言 → L2 兜底

**Dify 响应解析要点（`extractJsonFromAnswer`）：**
Dify 返回的 `answer` 字段可能混入 AI 思考文字或 Markdown 代码块标记，不能直接 `JSON.parse`。解析时需要兼容三种格式：
1. Markdown 代码块包裹：` ```json { ... } ``` ` → 剔除前后标记
2. 混入思考文字：`思考内容... { ... }` → 截取第一个 `{` 到最后一个 `}`
3. 纯净 JSON：直接解析

> 详细实现参见 `.trae/docs/guide/AI词语查询模块.md` 中的 `extractJsonFromAnswer` 方法和 `.trae/docs/guide/RestTemplate发起客户端请求流程.md` 中的解析流程。

### 5.6 AI 人设自定义工坊（创新加分项）

玩家可通过自然语言描述创建独特的 AI 对手：

```
玩家输入: "一个害怕被发现的胆小鬼，说话总是结结巴巴，还喜欢转移话题"
         │
         ▼
Dify persona-creator 工作流
         │
         ▼
输出 System Prompt:
"你是一个极其胆小的玩家，你非常害怕被人发现你是卧底。
说话时经常结巴（用省略号表示停顿），喜欢突然转换话题。
描述时故意模糊不清，但会假装很自信。
当有人质疑你时，你会紧张地转移焦点到其他人身上。"
```

- 创建的人设存入 `user_personas` 表
- 每月限创建 3 个（防滥用）
- 人设通过审核后上架到公共工坊，其他玩家可选用

## 六、房间管理器 (`game.manager.ts`)

### 6.1 设计
- 内存 Map `<gameId, GameRoom>` 存储活跃房间
- Redis 备份（用于服务重启恢复 + 横向扩展共享状态）
- 房间生命周期：
  - 创建时分配 UUID → 加入 Map + Redis
  - 游戏中：实时更新 Redis（发言/投票状态）
  - 结束后 5 分钟 → 清理 Map + Redis

### 6.2 GameRoom 结构

```typescript
interface GameRoom {
  gameId: string
  status: GameStatus
  players: RoomPlayer[]
  currentRound: number
  currentPhase: 'speaking' | 'voting'
  speakerQueue: number[]        // 发言顺序（slotIndex）
  civilianWord: string
  undercoverWord: string
  undercoverSlotIndex: number
  speeches: Speech[]           // 当前轮发言
  votes: Map<number, number>   // voterSlot → targetSlot
  createdAt: Date
  timerHandle?: NodeJS.Timeout
}
```

## 七、游戏回放系统（创新加分项）

### 7.1 设计思路
每局游戏的所有关键事件按时间线记录到 `game_replay_events` 表，结束后生成可交互的回放。

### 7.2 采集事件

```typescript
type ReplayEventType =
  | 'game_start'
  | 'round_start'
  | 'speech'          // { slotIndex, content, persona? }
  | 'vote_cast'       // { voterSlot, targetSlot }
  | 'elimination'     // { slotIndex, role }
  | 'round_end'
  | 'game_over'       // { winner, civilianWord, undercoverWord }
```

### 7.3 回放接口

```
GET /api/games/:id/replay → 返回完整事件时间线
```

前端渲染为时间轴播放器：
- 进度条拖动跳转到任意时刻
- 每条发言带玩家头像 + 人设标签
- 投票阶段展示票型动画
- 淘汰瞬间展示角色揭露
- AI 心理旁白（由 Dify 后处理生成，插入到关键节点）

## 十一、安全设计

1. **密码**：bcrypt(12 rounds) 哈希存储
2. **JWT**：Access Token 15min + Refresh Token 7d，使用 HTTP-only Cookie（可选）+ Authorization Header
3. **输入校验**：所有请求体使用 Zod Schema 严格校验
4. **速率限制**：`@fastify/rate-limit` 限制 API 调用频率
5. **CORS**：仅允许前端域名
6. **Helmet**：HTTP 安全头
7. **游戏作弊防护**：词语仅在服务端存储，不通过 WebSocket 明文传输给其他玩家
8. **SQL 注入**：Drizzle ORM 参数化查询

## 十二、部署架构

```
┌──────────┐    HTTPS/WSS     ┌─────────────┐
│  CDN     │◄────────────────►│   Nginx     │
│ (静态资源)│                  │  (反向代理)  │
└──────────┘                  └──────┬──────┘
                                     │
                              ┌──────▼──────┐
                              │  Fastify    │
                              │  (Node.js)  │
                              └──┬──────┬───┘
                                 │      │
                          ┌──────▼─┐ ┌──▼──────┐
                          │PostgreSQL│ │ Redis  │
                          └────────┘ └─────────┘
```

- Nginx：SSL 终结、静态资源代理、socket.io 反向代理（自动协商传输层）
- Fastify：单进程（初期），后续可 PM2 cluster 横向扩展
- Redis Pub/Sub：多进程/多实例间同步 WebSocket 消息
